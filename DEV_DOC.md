# Developer Documentation — Inception

This document describes how to set up, build, and manage the Inception infrastructure from a developer perspective.

## Prerequisites

- A Linux virtual machine (Debian/Ubuntu recommended)
- Docker Engine installed: `docker --version`
- Docker Compose plugin installed: `docker compose version`
- `make`, `openssl`, `sudo` available
- The domain `ghambrec.42.fr` must resolve to `127.0.0.1`:

```sh
echo "127.0.0.1 ghambrec.42.fr" | sudo tee -a /etc/hosts
```

## Configuration Files

### `srcs/.env`

Contains non-sensitive environment variables passed to containers via `env_file`:

```env
DOMAIN_NAME=<login>.42.fr
MYSQL_DATABASE=wordpress
MYSQL_USER=<db_user>
WP_TITLE=Inception
WP_DB_HOST=mariadb
WP_ADMIN_USER=<admin-user>
WP_ADMIN_EMAIL=<admin-user>@example.com
WP_USER=<wp-user>
WP_USER_EMAIL=<wp-user>@example.com
FTP_USER=<ftp-user>
GITEA_ADMIN_USER=<gitea-admin-user>
```

> Do not put passwords here. Use Docker secrets instead.

### `secrets/` directory

Generated by `make secrets`. Each file contains a random 32-character hex string:

```
secrets/db_user_password.txt
secrets/db_root_password.txt
secrets/wp_admin_password.txt
secrets/wp_user_password.txt
secrets/ftp_user_password.txt
secrets/gitea_admin_password.txt
```

Containers access secrets by reading `/run/secrets/<secret_name>` at runtime. This is more secure than passing passwords through environment variables (which can appear in `docker inspect` output).

## Building and Launching the Project

### Full build and start

```sh
make
```

Equivalent to:

```sh
make secrets                          # Generate secrets if missing
mkdir -p /home/ghambrec/data/db
mkdir -p /home/ghambrec/data/wordpress
mkdir -p /home/ghambrec/data/gitea
mkdir -p /home/ghambrec/data/gitea_config
docker compose -f srcs/docker-compose.yml up -d --build
```

### Build without starting

```sh
docker compose -f srcs/docker-compose.yml build
```

### Rebuild a single service

```sh
docker compose -f srcs/docker-compose.yml build nginx
docker compose -f srcs/docker-compose.yml up -d --no-deps nginx
```

## Managing Containers and Volumes

### Container lifecycle

| Command | Effect |
|---|---|
| `make up` | Build images and start all containers |
| `make stop` | Stop containers (state preserved) |
| `make down` | Stop and remove containers |
| `make clean` | Remove containers, volumes, and `/home/ghambrec/data/` |
| `make fclean` | Full reset: prune all images and delete secrets |
| `make re` | `fclean` + `all` |
| `make logs` | Follow all container logs |

### Useful Docker commands

```sh
# List containers
docker compose -f srcs/docker-compose.yml ps

# Execute a shell inside a running container
docker exec -it mariadb bash
docker exec -it wordpress bash
docker exec -it nginx bash

# Inspect a container
docker inspect mariadb

# View resource usage
docker stats

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect srcs_db
```

### Healthcheck (MariaDB)

The `mariadb` service has a Docker healthcheck that runs every 5 seconds. WordPress and Adminer wait for it to be healthy before starting:

```yaml
healthcheck:
  test: ["CMD-SHELL", "mariadb -u root --password=$(cat /run/secrets/db_root_password) -e 'SELECT 1'"]
  interval: 5s
  timeout: 3s
  retries: 10
  start_period: 10s
```

## How Each Service Is Initialized

### NGINX (`srcs/requirements/nginx/`)

- Base image: `debian:bookworm`
- A self-signed TLS certificate is generated during the image build with `openssl`
- `nginx.conf` is a template; `${DOMAIN_NAME}` is substituted at container startup via `envsubst`
- Only TLSv1.2 and TLSv1.3 are accepted
- Forwards PHP requests to `wordpress:9000` via FastCGI

### WordPress (`srcs/requirements/wordpress/`)

- Base image: `debian:bookworm`
- Installs `php-fpm`, `php-mysql`, and `wp-cli`
- `init.sh` downloads WordPress core and runs `wp core install` on first startup (guarded by checking for `wp-config.php`)
- Creates an admin user and a regular author user
- Installs and enables the `redis-cache` plugin
- Listens on `0.0.0.0:9000` (PHP-FPM)

### MariaDB (`srcs/requirements/mariadb/`)

- Base image: `debian:bookworm`
- `init.sh` generates an SQL init file that creates the database, the WordPress user, and sets the root password
- Uses `--init-file` flag to run the SQL on first `mysqld` start
- Listens on `0.0.0.0:3306`

### Redis (`srcs/requirements/bonus/redis/`)

- Base image: `debian:bookworm`
- Custom `redis.conf` with `bind 0.0.0.0`
- Acts as an object cache for WordPress via the `redis-cache` plugin

### FTP (`srcs/requirements/bonus/ftp/`)

- Base image: `debian:bookworm`
- Uses `pure-ftpd`
- `init.sh` creates the FTP user and sets the password from secrets
- Mounts the WordPress volume at `/home/wordpress`
- Passive mode ports: 30000–30009

### Adminer (`srcs/requirements/bonus/adminer/`)

- Base image: `debian:bookworm`
- Downloads `adminer.php` at build time and serves it with PHP's built-in server on port 8080

### Gitea (`srcs/requirements/bonus/gitea/`)

- Base image: `debian:bookworm`
- Downloads the Gitea binary at build time
- `init.sh` creates the admin user using Gitea's CLI on first run

## Data Persistence

All persistent data is stored on the host under `/home/ghambrec/data/`:

| Host Path | Docker Volume | Used By |
|---|---|---|
| `/home/ghambrec/data/db` | `db` | MariaDB database files |
| `/home/ghambrec/data/wordpress` | `wordpress` | WordPress files (themes, uploads, plugins) |
| `/home/ghambrec/data/gitea` | `gitea` | Gitea repositories and data |
| `/home/ghambrec/data/gitea_config` | `gitea_config` | Gitea configuration |

These are Docker named volumes configured with `driver: local` and `o: bind` to pin them to specific host paths. Data survives container restarts and even `docker compose down`. It is only removed by `make clean` or `make fclean`.

## Network Architecture

All containers are connected to a single bridge network named `inception`:

```
Host
 └─ port 443  ──► nginx ──► wordpress:9000 (FastCGI)
                        └──► mariadb:3306
                        └──► redis:6379
 └─ port 8080 ──► adminer
 └─ port 3000 ──► gitea
 └─ port 21   ──► ftp
```

Containers communicate by DNS name (service name). The host network (`network: host`) and `--link` flags are explicitly forbidden.

## Security Notes

- No passwords are stored in Dockerfiles or image layers
- All credentials use Docker secrets mounted at `/run/secrets/`
- The `secrets/` directory is git-ignored
- NGINX is the only entry point on port 443; all other services are internal except explicitly exposed bonus ports
- The `latest` tag is not used anywhere — all base images are pinned to `debian:bookworm`
